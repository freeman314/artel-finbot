# Сборка общих требований

Скилл детерминизирует один шаг конвейера артели: из пяти личных «желаний»
получить **один канонический общий артефакт**, который каждый может пересобрать
той же командой и проверить через PR. Это заменяет пять расходящихся личных
сводок: единый источник правды вместо форков.

Артефакт — **два файла в `common/`**:

- `common/prd.md` — то, с чем участники согласны (совпадения) и что предложено
  в одиночку (одиночные требования). Каркас будущего продукта.
- `common/contradictions.md` — места, где желания несовместимы. Каждое —
  отдельным блоком, выносится на голосование community.

Это **не PRD-спецификация**. Мы сводим желания, а не проектируем продукт.

## Прежде чем начать

- **Papкой чужого не трогай.** Читаешь `*/requirements.md` всех участников —
  это можно. Пишешь только в `common/`. Личные папки участников не редактируешь.
- **`common/` — по договорённости.** Результат сборки — всегда **черновик к
  голосованию**, а не «утверждённая истина». Ставь это в шапку обоих файлов.
  «Согласовано» у конкретного блока появляется только после голосования артели.
- **Не в одиночку коммить в `common/`.** Собрать и показать — да. Влить —
  по договорённости с артелью.
- **Идемпотентность.** При пересборке НЕ затирай проставленные людьми статусы
  голосования и итоги. Если старый `contradictions.md` уже есть — перечитай его,
  сохрани строки `**Статус:**` и `**Решение:**`, обнови только состав сторон,
  авторство и даты. Расхождение (противоречие исчезло/появилось новое) — отметь
  явно, не молча.

## Вход

Все `*/requirements.md`. Незаполненные (остался шаблон с тремя пустыми
разделами) — это **не** «требований нет». Помечай автора «нет данных (шаблон не
заполнен)» и не выдумывай за него.

## Шаг 1. Даты из git

Дата у требования — не выдуманная, а когда его текст реально попал в репозиторий:

```
git log --date=short --format='%ad  %h  %s' -- <папка>/requirements.md
```

Берётся дата коммита, в котором формулировка появилась. Если блок добавлен
позже остального файла (отдельным коммитом) — датируй его этим коммитом
(`git blame -L <строки> -- <файл>` покажет точно). Формат в тексте: `2026-07-13`.

## Шаг 2. Разложить по темам, а не по авторам

Структура общего документа — **по темам продукта**. Тогда совпадения сами
слипаются в один блок, а несовместимости оказываются рядом. Ориентир (не догма,
темы бери из реального материала):

источники данных и ввод · категоризация · мультивалютность · аналитика и суть
продукта · хранение, синхронизация, приватность · интерфейс доступа ·
стоимость · точность и проверяемость · автоматизация против ручного труда ·
персонализация и форк · предпринимательский/налоговый слой.

## Шаг 3. Классифицировать каждый тезис

Разбери требования на отдельные тезисы и каждый отнеси к одному из трёх:

1. **Совпадение** — одного хотят двое и больше. → `prd.md`, перечисли всех авторов.
2. **Одиночное** — воззвал один, остальные не против (или не высказались). →
   `prd.md`, один автор.
3. **Противоречие** — желания несовместимы (одному нужно X, другому не-X, либо
   границы взаимоисключающи). → `contradictions.md`.

Не сглаживай противоречия и не выбирай «правильную» сторону — противоречие
это ценный результат, ради него всё и затевается. Если сомневаешься, совпадение
это или противоречие, — оформи как мягкое противоключие со статусом «уточнить».

## Шаг 4. Формат `common/prd.md`

Заголовок файла — шапка «черновик к голосованию» + дата сборки + кто не сдал
требования. Дальше разделы по темам, внутри — блоки с пометкой `[авторы · дата]`:

```markdown
### Свои категории с иерархией
Категории задаёт пользователь; иерархия «супер → под», сворачивание при анализе.
— [Василий · 2026-07-10, Владимир · 2026-07-11]

### Крипта как источник (биржи по API, кошельки по сети)
Пока вручную (скриншоты → распознавание), хочется автоматизировать.
— [Владимир · 2026-07-11]
```

- Совпадение — перечисляй всех авторов, у каждого своя дата.
- Одиночное — один автор.
- Снятое требование не удаляй: `~~текст~~ — снято [автор · дата]`.

## Шаг 5. Формат `common/contradictions.md`

Каждое противоречие — отдельный блок: стороны с авторством, предложение по
разрешению, статус. «Проигравшая» формулировка после голосования не удаляется.

```markdown
### ⚠ Основной интерфейс доступа
- **A.** Android-приложение, десктоп не нужен. — [Илья · 2026-07-13]
- **B.** Отдельного приложения не надо, хватает телеграм-бота. — [Владимир · 2026-07-11]
- **C.** Дашборд + доступ через MCP-слой агентов. — [Василий · 2026-07-10]
- **Предложение по разрешению:** …
- **Статус:** на голосовании   ← | на голосовании | уточнить | решено
- **Решение:** —               ← заполняется после голосования, со ссылкой на MoM
```

## Чего в сборке не делать

- Не проектируй UI, вайрфреймы, сценарии по шагам, архитектуру, стек. Рано.
  Если это есть в личных требованиях — перенеси как **желание** автора, не
  разворачивай в спецификацию.
- Не выдумывай требований, которых не написали. Пустой автор — «нет данных».
- Не сглаживай и не «решай» противоречия за артель. Твоё дело — предложение,
  голос за community.
- Не раздувай. Каркас, а не двадцать страниц, которые не прочтут перед голосованием.

## Коммит

Собранное показывай, влитие в `common/` — по договорённости. Сообщение по-русски,
по существу: «сборка общих требований: черновик к голосованию от 2026-07-18».
